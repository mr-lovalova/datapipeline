# Python Integrations

`datapipeline.integrations.ml` demonstrates how to reuse the runtime from
application code:

- `VectorAdapter.from_project(project_yaml, output_id=...)` – load one
  definition, then stream vectors or row dicts. Datasets with `split` require a
  concrete fold output such as `holdout.train`; unsplit datasets omit
  `output_id`.
- `stream_vectors(project_yaml, output_id=..., limit=...)` – yield `(sample_key,
  feature_vector)` pairs from already-current artifacts. Unlike `jerry serve`,
  it does not prepare stale artifacts or persist output. Split outputs use
  their fold's fitted scaler; unsplit datasets use the standard scaler when
  scaling is configured. The stored series artifacts remain unscaled.
- `iter_vector_rows` / `collect_vector_rows` – handy for Pandas or custom sinks.
- `dataframe_from_vectors` – eager helper that returns a Pandas DataFrame
  (requires `pandas`).
- `torch_dataset` – builds a `torch.utils.data.Dataset` that yields tensors. In
  the workspace generated by `jerry demo init`, see
  `demo/scripts/run_torch.py` for usage (install `jerry-thomas[ml]` first).

Every helper accepts the same `output_id`. It is required when `dataset.split`
is configured and rejected for an unsplit dataset.
